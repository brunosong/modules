# Flyway 스터디 노트

## 여러 서비스가 하나의 DB를 공유할 때 생기는 문제들

### 1. 누가 뭘 바꿨는지 모른다 (추적 불가)

```
월요일: chatting팀이 users 테이블에 컬럼 추가
화요일: agent팀이 같은 테이블에서 컬럼명 변경
수요일: chatting 서비스 터짐 → "누가 건드렸어?"
```

DB에 직접 붙어서 DDL을 실행하면 기록이 안 남는다. git blame처럼 "이 컬럼 누가 왜 추가했지?"를 추적할 방법이 없다.

### 2. 환경마다 스키마가 다르다 (환경 불일치)

```
개발 DB:  users 테이블에 컬럼 15개 (개발자 A가 수동으로 추가)
스테이징: users 테이블에 컬럼 13개 (까먹고 안 넣음)
운영:     users 테이블에 컬럼 14개 (일부만 반영)
```

"내 로컬에서는 되는데요?" 의 DB 버전. 수동 관리하면 환경이 점점 drift(표류)한다.

### 3. 배포 순서에 의존하게 된다 (암묵적 의존성)

```
agent 서비스: users.profile_url 컬럼이 있다고 가정하고 코드 작성
chatting 서비스: 그 컬럼을 추가하는 ALTER TABLE을 아직 안 실행함
```

agent를 먼저 배포하면 터진다. 하지만 이 의존관계는 코드 어디에도 명시되어 있지 않다.

### 4. 동시에 같은 테이블을 건드린다 (충돌)

```
chatting팀 PR: ALTER TABLE users ADD COLUMN phone VARCHAR(20);
agent팀 PR:    ALTER TABLE users ADD COLUMN phone VARCHAR(50);
```

둘 다 따로 테스트하면 통과한다. 하지만 둘 다 merge되면 하나가 실패한다. 코드 충돌은 git이 잡아주지만, 스키마 충돌은 아무도 안 잡아준다.

### 5. 롤백이 안 된다 (되돌리기 불가)

변경 전 상태가 기록되어 있지 않으면 복원 자체가 추측이 된다.

### 6. 코드와 스키마가 따로 논다 (버전 불일치)

코드는 git 태그로 버전 관리가 되는데, DB 스키마는 어떤 시점에 어떤 상태였는지 알 수 없다.

---

## Flyway가 해결하는 것

| 문제 | Flyway의 해결 |
|------|--------------|
| 누가 바꿨는지 모름 | SQL 파일이 git에 남음 → blame, PR 이력 |
| 환경마다 다름 | 어디서 실행해도 같은 순서로 같은 SQL 적용 |
| 암묵적 의존성 | 버전 순서가 명시적 → 선후관계 보장 |
| 스키마 충돌 | PR에서 파일 충돌로 사전 감지 |
| 롤백 불가 | 이력 테이블에 뭘 언제 적용했는지 기록 |
| 코드-스키마 불일치 | 앱 기동 시 자동 검증 → 불일치면 기동 실패 |

> **"DB에 직접 DDL을 치는 건 git 없이 코드를 관리하는 것과 같다."**

## DB 스키마 관리 방식 비교

### 1. 중앙 집중형 (채택한 방식) — 독립 flyway 모듈

```
flyway/           ← 모든 서비스의 DDL이 여기에 모임
chatting/         ← 비즈니스 코드만
agent/            ← 비즈니스 코드만
```

- 장점: PR 리뷰 한 곳, 충돌 감지 쉬움, DBA가 보기 편함
- 단점: 서비스 배포와 스키마 배포 타이밍이 분리됨
- 적합: DB가 하나이고, 스키마 변경을 엄격히 통제하고 싶을 때

### 2. 각 서비스에 내장

각 서비스가 기동할 때 자기 마이그레이션을 실행하는 방식.

- 장점: 서비스와 스키마가 함께 배포됨 (코드-스키마 동기화)
- 단점: DB가 하나인데 여러 서비스가 각자 flyway를 돌리면 race condition 발생 가능. 서비스 A가 서비스 B 테이블을 건드리는 걸 막기 어려움
- 적합: 서비스별 DB가 분리된 MSA 환경

### 3. Liquibase (Flyway 대안)

XML/YAML/JSON으로 변경셋을 정의하는 방식.

- 장점: rollback 내장, diff 명령으로 DB와 코드 차이 자동 감지, DB 벤더 독립적
- 단점: SQL 직접 작성보다 장황, 러닝커브
- 적합: 멀티 DB 벤더 지원이 필요하거나 rollback을 자동화하고 싶을 때

### 4. SchemaHero / Atlas (선언형)

"현재 상태"를 선언하면 도구가 마이그레이션을 자동 생성하는 방식.

- 장점: 마이그레이션 파일 직접 안 써도 됨, GitOps 친화적
- 단점: Java/Spring 생태계와 통합이 약함, 복잡한 데이터 마이그레이션에 부적합
- 적합: K8s 기반 인프라, Go/Node 생태계

### 방식 비교 요약

| 방식 | DB 공유 | 통제력 | 복잡도 | 현재 상황 적합도 |
|------|---------|--------|--------|-----------------|
| **중앙 flyway 모듈** | O | 높음 | 낮음 | **최적** |
| 서비스 내장 flyway | X (DB 분리 시) | 중간 | 낮음 | 부적합 |
| Liquibase | O | 높음 | 중간 | 가능하나 과함 |
| Atlas/선언형 | O | 높음 | 높음 | 생태계 불일치 |

## 실제 대규모 프로젝트에서는?

### 네이버/카카오급 (DB 하나, 팀 여럿)

중앙 집중형과 거의 동일하되 추가로:
- DBA 승인 게이트: PR merge만으로 바로 적용 안 됨. DBA가 별도 승인 후 실행
- CI에서 dry-run: PR 올리면 테스트 DB에 먼저 적용해보고 결과 리포트
- 슬로우 쿼리 체크: ALTER TABLE이 대용량 테이블에 락을 걸지 않는지 자동 검증

### 대형 MSA (서비스 수십~수백 개)

- 서비스별 DB 분리 + 각 서비스에 flyway 내장
- 공유 데이터는 이벤트/API로 동기화, 직접 다른 서비스 테이블 접근 금지
- 스키마 레지스트리로 계약(contract) 관리

### 중간 규모 (DB 하나, 서비스 3~10개)

가장 현실적인 조합: **중앙 flyway 모듈 + 도메인별 디렉토리 + PR 리뷰 + CI dry-run**

## 중앙 집중식의 장점 (상세)

### 1. 변경이 한눈에 보인다

DB에 대한 모든 변경이 하나의 디렉토리에 쌓이기 때문에, PR 목록만 봐도 "지금 DB에 무슨 일이 일어나고 있는지" 전체 그림이 보인다. 각 서비스에 흩어져 있으면 내가 모르는 사이에 다른 팀이 같은 테이블을 건드리고 있어도 알 방법이 없다.

### 2. 충돌을 배포 전에 잡는다

같은 디렉토리에 파일이 모이니까, 같은 테이블에 대한 변경이 동시에 들어오면 PR 리뷰 단계에서 발견된다. 서비스별로 분리되어 있으면 각자의 리뷰는 통과하고, 운영 DB에서 충돌이 터져야 비로소 알게 된다.

### 3. 리뷰어가 맥락을 갖는다

flyway 모듈의 리뷰어는 DB 변경 전체 흐름을 계속 보는 사람이다. "지난주에 이 테이블 구조 바꿨는데 또 바꾸는 거야?", "이 인덱스 지난달에 추가한 거랑 중복 아니야?" — 이런 판단은 전체 이력을 보는 사람만 할 수 있다.

### 4. 실행 순서가 보장된다

하나의 flyway가 하나의 버전 히스토리를 관리하므로, A 다음에 B가 실행되어야 한다는 순서가 명확하다. 서비스별로 각자 flyway를 돌리면, 누가 먼저 기동하느냐에 따라 실행 순서가 달라질 수 있다.

### 5. 새로운 환경을 똑같이 만들 수 있다

개발자가 새로 합류하거나 테스트 DB를 새로 만들어야 할 때, `./gradlew :flyway:bootRun` 하나면 운영과 동일한 스키마가 만들어진다. 서비스별로 흩어져 있으면 순서를 사람이 알아야 한다.

### 6. DB 스키마에 대한 git log가 생긴다

```bash
git log --oneline flyway/src/main/resources/db/migration/
```

이 디렉토리의 git log가 곧 DB의 변경 역사다. 커밋, PR, 리뷰 코멘트까지 전부 추적된다.

### 7. CI에서 사전 검증을 걸 수 있다

flyway 모듈이 하나니까 CI 파이프라인도 하나만 만들면 된다. 서비스별로 흩어져 있으면 검증을 서비스마다 각각 만들어야 하고, 서비스 간 합쳤을 때의 문제를 잡을 수 없다.

### 8. 운영 사고 때 원인 파악이 빠르다

```bash
git log --since="3 days ago" flyway/src/main/resources/db/migration/
```

한 곳에 모여 있으니까 최근에 뭐가 바뀌었는지 한 번에 확인된다.

## 소유권이 불분명할 때의 마이그레이션 구성

테이블 소유권이 명확하지 않은 현실적인 상황에서의 접근법.

### 추천: 한 폴더 + 파일명 태그

```
db/migration/
├── V202604210001__create_users.sql
├── V202604210002__CHAT_create_chat_room.sql
├── V202604210003__CHAT_add_phone_to_users.sql
├── V202604210004__AGENT_create_agent_config.sql
└── V202604210005__AGENT_add_model_pref_to_users.sql
```

SQL 파일 헤더에 누가, 왜 만들었는지 남기기:

```sql
-- ============================================================
-- Author  : brunosong (chatting팀)
-- Why     : 채팅방에서 사용자 전화번호 표시 필요
-- Related : JIRA-1234
-- ============================================================
ALTER TABLE users ADD COLUMN phone VARCHAR(20);
```

> **테이블의 "소유권"을 정하려고 하지 말고, 변경의 "책임자"만 명확히 해라.**

## 버전 채번 방식 비교

### 1. 순차 번호 (`V1`, `V2`, `V3`)
- 장점: 단순, 직관적
- 문제: 여러 사람이 동시에 작업하면 번호 겹침

### 2. 타임스탬프 (`V202604211430`) — 추천
- 장점: 충돌 확률 거의 0, 만든 시점 바로 알 수 있음
- 단점: 번호가 길어서 한눈에 순서 파악이 좀 느림
- **실무에서 가장 많이 사용**

### 3. 날짜 + 순번 하이브리드 (`V20260421_001`)
- 장점: 날짜별로 묶이고, 같은 날 안에서 순서도 보임
- 단점: 같은 날 같은 순번 가능성은 여전히 있음

### 비교 요약

| 기준 | 순차 번호 | 타임스탬프 |
|------|----------|-----------|
| 충돌 가능성 | 높음 (팀 커지면 필연) | 거의 0 |
| 커뮤니케이션 비용 | "다음 번호 몇이야?" | 없음 (지금 시각 쓰면 됨) |
| 만든 시점 파악 | 불가 | 즉시 가능 |
| 실수 가능성 | 번호 빠뜨림/중복 | 시각만 찍으면 됨 |

분 단위 (`V202604211430`)면 현실적으로 충분하다.

## application.yml 옵션 설명

### `spring.flyway.enabled`
Flyway 자동 실행 on/off. `false`로 하면 앱 기동 시 마이그레이션을 실행하지 않는다.

### `locations`
SQL 파일을 스캔할 경로 목록. 여기에 없는 경로의 SQL은 무시된다.

### `baseline-on-migrate`
기존에 이미 테이블이 있는 DB에 Flyway를 처음 도입할 때 필요. `true`면 자동으로 baseline을 잡고 진행. `false`면 히스토리 테이블이 없을 때 에러.
- 신규 DB: 상관없음
- 기존 운영 DB에 Flyway 도입: **반드시 true**
- 도입 완료 후: false로 바꿔도 됨

### `validate-on-migrate`
이미 적용된 SQL 파일이 변경되지 않았는지 checksum 비교. **반드시 true로 유지**. 꺼져 있으면 적용된 파일을 몰래 수정해도 경고 없이 넘어간다.

### `out-of-order`
V3까지 적용된 상태에서 V2.5를 추가하면 어떻게 할 것인가?
- `false`: 에러 — 과거 버전 끼워넣기 금지
- `true`: 허용 — 늦게 merge된 PR의 SQL도 실행
- 여러 팀이 동시에 작업하면 **true가 현실적**

### `clean-disabled`
`false`로 하면 `flyway clean` 명령이 DB 전체를 날린다. **절대 false로 바꾸지 말 것**.

### `connect-retries`
DB 연결 실패 시 재시도 횟수. Docker 환경에서 앱이 DB보다 먼저 뜰 때 유용.

## 참고: build.gradle 의존성

Flyway가 동작하려면 `spring-boot-starter-jdbc`가 필요하다. `spring-boot-starter`만으로는 DataSource 빈이 생성되지 않아 마이그레이션이 실행되지 않는다.

```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-jdbc'  // starter만으로는 안 됨
    implementation 'org.flywaydb:flyway-core'
    implementation 'org.flywaydb:flyway-mysql'
    runtimeOnly 'com.mysql:mysql-connector-j'
}
```
